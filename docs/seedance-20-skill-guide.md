**# 1. 这到底是「什么东西」？**

- **不是** 一个单独的安装在你电脑里的视频软件。  

- **是** 一套给 **AI 助手**读的「说明书 + 方法论」：教模型怎么写 **Seedance 2.0** 可用的 **提示词（Prompt）**、怎么分镜、怎么规避版权与审查、怎么排查生成失败等。  

- **通俗理解**：就像给编剧、导演用的 **分镜与制作手册**；真正「出片」仍要通过 **字节跳动 / 火山引擎** 侧的产品或 **API**（例如你项目里用的 **Ark 内容生成接口**）。

**四模态（Quad-modal）**：同一套方法论覆盖四种典型任务（技能文档中的说法）：

| 缩写 | 含义 | 典型输入 |

|------|------|----------|

| **T2V** | 文生视频 | 纯文本描述 |

| **I2V** | 图生视频 | 首帧图 + 文本（你流水线里 Step 7 接近这种） |

| **V2V** | 视频参考 / 改视频 | 视频 + 文本（改什么、保留什么） |

| **R2V** | 多参考生视频 | 多图 / 多模态参考 + 文本（@Tag 体系） |

## **3. 详细使用指南（怎么「用好」这套技能）**

### **3.1 推荐的日常流程**

1. **有完整剧本 / 产品 Brief** → 可先走 **seedance-interview**，把信息补全成「可拍清单」。
2. **写每一条具体生成指令** → 用 **seedance-prompt** 的 **五层结构** 搭句（见下）。
3. **涉及真人脸、大牌、动漫角名、知名场面** → 必须过 **seedance-copyright**，改成「类型 / 气质」描述而非点名 IP。
4. **画面不对劲** → **seedance-troubleshoot** + 相关专精子技能（运镜 / 光 / 角色 / 动效）。
5. **要做系列片、广告模板** → **seedance-recipes**。

### **3.2 五层 Prompt 结构（seedance-prompt 核心）**

按顺序写，且 **把重要信息放前面**（前 20～30 个词权重更高）：

1. **SUBJECT**：主体是谁 / 什么
2. **ACTION**：主要动作（**原则：一句话里主要动词别堆太多**）
3. **CAMERA**：景别、角度、运动方式、速度感
4. **STYLE**：少量高信号风格词（电影语言优于堆砌形容词）
5. **SOUND**：环境音、音效、音乐气质（若你的 API 支持音频相关参数再对应填写）
6. **CONSTRAINTS**：必须保持什么、不要什么（一致性、禁忌）

**经验公式（技能里称「6 段场域公式」）**：

[镜头类型] + [主体] + [动作] + [风格] + [镜头运动] + [声音线索]

### **3.3 @Tag 与参考图（概念层）**

技能强调：**每个 @ 引用要有明确角色**（例如：谁当主体、谁当首帧、谁当风格参考）。并区分：

- **参考**：学其运镜 / 气质，生成新画面
- **编辑**：在原视频上改内容（不同写法会触发不同模型行为）

**输入上限（技能中的「Rule of 12」）**：图片至多 9、视频至多 3、音频至多 3、总参考文件至多 12（具体以当下平台 / API 文档为准）。

> ***与你当前代码的关系**：你项目里的* `seedance_video.py` *用 **首帧 + 可选多张参考图** 走 Ark，更接近技能里 **I2V + 多参考** 的子集；**不一定**在 API 层暴露 @Tag 语法，但 **Prompt 写法**可以直接借鉴五层与 @ 思路（把「谁参考谁」写进自然语言）。*

### **3.4 「委托层级」Delegation Levels（控制用多长的说明）**


| **层级** | **长度量级**      | **适用**                           |
| ------ | ------------- | -------------------------------- |
| L1     | 不超过约 30 词     | 模型「本来就懂」的日常品类，放手让模型编排            |
| L2     | 约 30～100 词    | **最常用**：主体 + 环境 + 一个镜头注 + 一个风格锚点 |
| L3     | 约 100～300 词   | 按时间段写：`0-3s / 3-7s / ...`        |
| L4     | 约 300～1000+ 词 | 打斗、对口型、精密产品演示等「全盘调度」             |


### **3.5 合规（seedance-copyright）——产品经理必看**

技能文档描述 **2026 年初** 以来平台侧对 **知名 IP、真人脸、流媒体原创** 等加强拦截；要点：

- **不要**在 Prompt 里直接写 **真实明星姓名、具体漫威 / 日漫角色名、Netflix 剧名式描述** 等（宜用「架空角色」「某类超英气质」替代）。
- **不要**要求 **换脸成某名人**、复刻 **带商标的特写**。
- **深伪 / 未授权肖像** 属于高风险表述，应一律避免。

> ***对你项目的含义**：Step 6 / 7 里脚本若从 LLM 来，要在 **进 Seedance 之前** 做一轮 **敏感词 / IP 替换** 或与人工审核流程结合，否则容易 **任务失败或无声降级**。*

## **4. 与本项目 manga-pipeline 的现状**

### **4.1 你已经「接上了」什么？**


| **能力**                       | **项目中的位置**                         | **说明**                          |
| ---------------------------- | ---------------------------------- | ------------------------------- |
| **Seedance 2.0 API（火山 Ark）** | `seedance_video.py`                | 创建任务、轮询、下载 MP4；首帧 + 多参考图 base64 |
| **作为流水线一步**                  | `steps/step_07_generate_videos.py` | 并行多镜头；读 `video_prompts` 与首帧路径   |
| **环境与模型**                    | `.env`、`SEEDANCE_MODEL` 等          | 与文档中「模型 ID 需与账号一致」一致            |


也就是说：**API 与工程集成已经可行且已在用**。技能包主要增强的是 **上游 Prompt 质量与合规**，而不是替代 API。

### **4.2 技能包 不能 直接替你做什么？**

- 不会自动改你仓库里的 Python 代码。
- 不会替代 **火山引擎控制台** 的计费、配额、地域限制。
- 技能里部分能力（如 **V2V、严格口型、复杂 @Audio**）是否与你当前 **Ark JSON 字段** 一一对应，需要以 **[官方文档](https://www.volcengine.com/docs/82379/1520757?lang=zh)** 为准逐步对齐。

## **5. 整合进本项目的可行性结论**

### **5.1 整体结论**


| **维度**                    | **可行性** | **说明**                                                                        |
| ------------------------- | ------- | ----------------------------------------------------------------------------- |
| **人工工作流**                 | **高**   | 团队在写分镜 / 视频 Prompt 时，把 seedance-20 当 **编剧手册** 用即可，零改代码。                       |
| **AI 辅助写 Prompt（Cursor）** | **高**   | 安装技能后，在 Cursor 里针对 `steps/`、`seedance_video.py` 相关需求 **@ 技能或复制 SKILL 要点** 即可。 |
| **全自动「技能进代码」**            | **中**   | 可在 Step 6 输出侧加 **模板化五层结构**、长度分段、或调用 LLM 做「Prompt 编译」；工作量中等，需测试。               |
| **合规自动化**                 | **中～高** | 可对 `video_prompts` 做 **规则 + LLM 二次审查**（对照 seedance-copyright 列表）；要平衡误杀与漏网。    |


### **5.2 建议的「渐进式」整合路线**

1. **零代码**：在团队文档中链接本文 + 官方 Ark 文档；编剧用五层结构写 `video_prompt` 字段。
2. **轻量代码**：在 `step_06_video_prompts` 后增加 **Prompt 格式化**（拼五层前缀、禁止词过滤、长度截断）。
3. **中等投入**：增加可选步骤 **「Prompt 审核 / 重写」**（仅当检测到 IP 词或高冲突场景）。
4. **重度**：对接 **V2V / 音视频多模态**（需扩展 `seedance_video._build_content` 与参数，严格对照 Ark 最新 schema）。

### **5.3 风险与注意点**

- **API 与 App 能力不完全等价**：技能面向「理想产品能力」；你实际用的是 **Ark 任务接口**，以返回值与错误码为准。
- **版权与审查演化快**：技能内 **Feb 2026** 背景仅供理解趋势；上线业务要以 **最新用户协议与区域法规** 为准。
- **成本**：更长的 L3 / L4 Prompt + 多参考图通常意味着 **更多重试与计费**，需在流水线里保留 **并发上限与超时**（你已有 `video_concurrency`、`max_wait_sec` 等）。
